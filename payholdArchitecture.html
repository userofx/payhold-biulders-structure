Mission

human process into programmable infrastructure.

The first implementation is escrow. Escrow is not the destination — it is the trust kernel. The long-term vision is to become the infrastructure layer that sits between people, assets, money, logistics, and agreements across Africa.

The problem

Four trust gaps killing African commerce.

Every failed transaction in Africa comes back to one of four categories. PayHold is systematically designed to close all four.

Gap 1
Payment risk
Buyer pays and never receives goods. Seller ships and never receives payment. This is the default assumption in Nigerian e-commerce.
Gap 2
Logistics risk
Goods are damaged. Deliveries disappear. Tracking is unreliable. No accountable chain of custody exists.
Gap 3
Identity risk
Counterparties are unknown. Reputation systems are weak or nonexistent. Anyone can be anyone online.
Gap 4
Asset transfer risk
Real estate. Vehicles. High-value goods. Future inheritance. Massive asset transfers happen with no neutral infrastructure.

Phase 2 deep-dive

The 45-minute hold window.

This is infrastructure logic, not UI logic. When a package is created, the system does not immediately assign the nearest dispatcher. It waits and optimises.

The core insight: Traditional logistics finds a rider then delivers a package. PayHold finds movement already happening and attaches a package to it. A driver going from the airport to town earns for a delivery along that route without changing their plans.
How the hold window works
T = 0:00
Package created, hold window opens
Transaction moves to MATCHING_DISPATCHER state. Redis sets a 45-minute TTL key. The scoring engine begins evaluating nearby dispatchers every 90 seconds.
T = 0:00 – 44:00
Scoring loop runs every 90 seconds
Each dispatcher is scored on: route overlap % (35%), reputation score (25%), ETA to pickup (25%), acceptance rate history (15%). Best scorer is the candidate but not yet assigned — the loop keeps watching for a better match.
T = 44:00
Assignment or escalation
If a candidate scores above threshold → assign immediately. If no candidate meets threshold → escalate to manual pool or extend window by 15 minutes (once only).
T = 45:00
Hard assignment
Best available dispatcher is assigned. If no dispatcher exists at all → transaction moves to EXPIRED and buyer is refunded. This hard cutoff prevents zombie transactions.
Technical implementation
// src/services/dispatcher/holdWindow.ts interface DispatcherScore { dispatcherId: string; routeOverlap: number; // 0-1 reputationScore: number; // 300-1000 etaMinutes: number; acceptanceRate: number; // 0-1 composite: number; // weighted total } const WEIGHTS = { routeOverlap: 0.35, reputationScore: 0.25, etaMinutes: 0.25, acceptanceRate: 0.15, }; const HOLD_WINDOW_MS = 45 * 60 * 1000; const SCORE_INTERVAL = 90 * 1000; // re-score every 90s const MIN_THRESHOLD = 0.65; // assign early if score ≥ 0.65 async function startHoldWindow(transactionId: string) { // Store in Redis with 45min expiry await redis.set( `hold:${transactionId}`, 'active', 'PX', HOLD_WINDOW_MS ); // BullMQ job: score loop + hard deadline await dispatchQueue.add('scoreLoop', { transactionId }, { repeat: { every: SCORE_INTERVAL }, jobId: `score-${transactionId}`, }); await dispatchQueue.add('hardAssign', { transactionId }, { delay: HOLD_WINDOW_MS, jobId: `assign-${transactionId}`, }); }
Why Redis + BullMQ
Redis TTL
The hold window clock
Redis TTL keys act as the canonical timer. If the server restarts, the TTL is still ticking. No in-memory timers that vanish on crash.
BullMQ
The scoring loop
Repeating jobs every 90 seconds. The hard-assign job fires at T=45:00 regardless of what else is happening. Retryable, observable, monitorable.
Geospatial
Route overlap scoring
Redis GEOADD + GEORADIUS for dispatcher proximity. Route overlap calculated against declared origin-destination. This is the critical scoring signal.

Trust scoring

The reputation engine.

PayHold's most defensible long-term asset. Every transaction scores buyers, sellers, and dispatchers. Scores compound over time — a new entrant starts at 500, an active user builds to 1000.

Three separate score identities
Role Top signal Decay Suspension threshold
Buyer Transaction completion (30%) · Release speed (25%) 2pts/month inactive Below 300
Seller Fulfilment rate (35%) · Dispute outcomes (25%) 3pts/month inactive Below 300 → manual review
Dispatcher On-time delivery (35%) · Package integrity (30%) 5pts/week inactive Below 400 → removed from pool
Tier thresholds
New — 300 to 499
Restricted access
Escrow creation blocked. Can browse but not transact. Complete identity verification to unlock 500+.
Verified — 500 to 699
Standard access
Escrow creation enabled. Standard limits and 72hr dispute SLA.
Trusted — 700 to 849
Priority access
Higher limits, 3-day→1-day hold period, 24hr dispute SLA, real estate escrow access.
The 850+ unlock matters for Phase 5: API access for B2B integrations requires elite-tier status. This creates a natural quality filter — only verified, high-volume, low-dispute actors become API partners. The trust system gates the business model.

Architecture decisions

What we decided and why.

✓
Kobo-only amounts: All monetary values stored as integers in kobo. No floats anywhere in the codebase. A ₦5,000 transaction is 500000 kobo. Prevents every floating-point rounding error in financial history.
✓
Monolithic first, microservices later: Start with a single Express app. Extract services to microservices only when a specific scaling bottleneck demands it. Premature microservices kill early-stage teams.
✓
AsyncStorage for MVP, Paystack deferred: Mobile app uses AsyncStorage for Phase 1 MVP. Paystack on mobile integrated after the backend payment flow is stable and tested.
✓
Schema supports all phases from day one: Prisma schema includes nullable Phase 2+ fields. No breaking migrations when Phase 2 launches. Plan the data model for the 5-year vision, build the features in sequence.
!
BullMQ + Redis required for Phase 2: The hold window needs persistent timers (Redis TTL) and retryable job queues (BullMQ). Add Redis to Railway before beginning Phase 2 work. Do not attempt the hold window with in-memory timers — they vanish on restart.
✗
No blockchain for vaults: PayHold Vaults are programmable custody on fiat rails, not smart contracts. Blockchain adds complexity, regulatory friction, and UX barriers without adding meaningful trust guarantees that a well-audited database state machine doesn't already provide.
✗
No dispatcher UI in Phase 1: The dispatcher network (Phase 2) is not touched until Phase 1 has 1,000 active transactions. Route density requires transaction volume. Transaction volume requires Phase 1 to be solid.
